## Apr 29, 2026

### `clientOrderId` echoed on every WebSocket `orderEvent`

Integrators can now reconcile WebSocket frames back to the originating `POST /orders` request without waiting on the HTTP response.

**What's new:**
- When you submit an order with a `clientOrderId`, every subsequent `orderEvent` for that order â€” `PLACEMENT`, `UPDATE`, `CANCELLATION`, and the `MINED` / `FAILED` settlement frame â€” echoes the same `clientOrderId` value.
- The field is omitted (not `null`) when the originating order had no client id.
- Counterparty ids (`takerOrderId`, `makerMatches[].orderId`) are intentionally **not** resolved â€” only the recipient user's own order maps to its client id.

**Why it matters:**
- The HTTP `POST /orders` path blocks until 1-conf settlement (`settlementStatus: MINED`). Previously, clients could not tie inflight WebSocket frames to a specific request id until the HTTP call returned.
- With this change, your client can fire an order, listen on `subscribe_order_events`, and reconcile every lifecycle event by `clientOrderId` immediately â€” placement, partial fills, cancellation, and on-chain settlement.

**Event payload note:**
- OME shapes carry `eventId: number` (monotonic SERIAL from the matching engine).
- Settlement shapes carry `eventId: string` in the form `settlement:<tradeEventId>`.
- These are separate types â€” keep your client typing distinct between the two.

**Documentation:** [WebSocket Events: subscribing to order events](/developers/websocket-events#subscribing-to-order-events) | [`orderEvent` payload reference](/developers/websocket-events#orderevent)

### WebSocket Order Events

Authenticated WebSocket clients can now subscribe to the CLOB order lifecycle with `subscribe_order_events`.

**New realtime stream:**
- Emits `orderEvent` messages for both off-chain matching-engine updates and on-chain settlement results
- OME events cover placement, updates, and cancellations
- Settlement events cover mined and failed transaction outcomes
- Auth failures are surfaced on the `exception` channel
- Clients should resubscribe on reconnect and treat OME and settlement events as independent streams

**Documentation:** [WebSocket Events](/developers/websocket-events#subscribing-to-order-events)

## Apr 2, 2026

### Programmatic API: Partner Integrations, Delegated Signing & FOK Orders

Major new capability for partners building on Limitless. The Programmatic API enables platform integrations with sub-account management, server-side order signing, and both GTC and FOK order types â€” all through HMAC-authenticated scoped tokens.

**New API endpoints:**
- `POST /auth/api-tokens/derive` â€” Create scoped HMAC tokens with granular permissions (`trading`, `account_creation`, `delegated_signing`)
- `GET /auth/api-tokens/capabilities` â€” Check partner capability configuration
- `GET /auth/api-tokens` â€” List active tokens
- `DELETE /auth/api-tokens/:tokenId` â€” Revoke a token
- `POST /profiles/partner-accounts` â€” Create sub-accounts with server-managed wallets or EOA verification

**Delegated signing:**
- Partners with the `delegated_signing` scope can submit unsigned orders on behalf of sub-accounts
- The server signs orders using Privy-managed wallets â€” no private key management needed on the partner side
- Both **GTC** (limit) and **FOK** (market) order types are supported for delegated orders
- FOK orders use `makerAmount` (USDC to spend for BUY, shares to sell for SELL) instead of `price` + `size`

**Recommended architecture:**
- Store HMAC credentials on your backend â€” never expose them to frontends
- Use the SDK server-side to sign partner-authenticated requests
- Expose only your own app-specific endpoints to the frontend

**Documentation:** [Programmatic API guide](/developers/programmatic-api) | [Authentication](/developers/authentication#scoped-api-tokens-hmac) | [API Tokens reference](/api-reference/api-tokens/derive-token)
